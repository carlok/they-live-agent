## REFERENCE VISION

The core visual inspiration is a blend between **They Live** (1988) and the **Terminator 2 bar scene** (1991): Arnold's POV as his system scans a biker and generates a rapid, cold, functional readout — height, weight, clothing sizes — all to answer a single operational question: *does this individual have what I need?* The goal of course if having fun, shoot, blast the face (in AR) of your friends or the like. :)

The mechanic we are building inverts this completely.

In T2, the readout is **accurate, mechanical, purposeful**.
In THEY LIVE: FIELD AGENT, the readout is **probabilistic, unreliable, and partly absurd** — because the Signal is unstable, the database is patchy, and the aliens have been seeding false human cover data for decades.

The player must read a HUD overlay on a real person and decide:
- Is this data suspicious, or just the noise of a broken system?
- Is this person alien or human — or is that even the right question?

The overlay is the **primary deduction surface** of the game.

---

## DESIGN PRINCIPLES FOR THE OVERLAY

1. **Plausible + absurd in the same breath.** The overlay must feel like a real system that is *slightly broken*. Some fields are mundane and believable. Some are surreal. The player never knows which category to trust.

2. **Narrative before numbers.** Every field is worded as if generated by a vast alien-infiltrated bureaucracy. Dry. Procedural. Occasionally ominous. The tone does the storytelling.

3. **No face data. No biometrics.** For real-world privacy and ethical design reasons (see Section 8), the overlay never references facial recognition, identity, or real biometric data. All fields are *invented by the system*, not sourced from the real person.

4. **Player literacy arc.** Early sessions: the overlay feels random and unreadable. By mid-game: the player has learned patterns — certain field combinations correlate with alien status. This is the Minesweeper layer: the overlay is the *numbered cell*.

5. **Variability is the mechanic.** No two scans look identical. Fields are drawn from a weighted pool and populated procedurally. The player is always reading, never pattern-matching a memorized screen.

## THE OVERLAY — VISUAL STRUCTURE

The overlay appears as a **translucent HUD panel** anchored above and slightly to the side of a detected person. It is styled as a degraded, cold-war-era computer terminal: scan lines, glitch artifacts, fields that type-in rapidly then stabilize.

Visual anatomy (top to bottom):

```
┌─────────────────────────────────────┐
│  ◈ SIGNAL LOCK — CONTACT #4         │  ← contact counter for the session
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  THREAT CONFIDENCE:  ██░░░░  34%    │  ← core deduction score (always shown)
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  [FIELD_01]         [VALUE_01]      │
│  [FIELD_02]         [VALUE_02]      │
│  [FIELD_03]         [VALUE_03]      │
│  [FIELD_04]         [VALUE_04]      │  ← 3-6 fields, drawn from pool
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  CLUSTER SIGNAL:    ▲ +2 NEARBY     │  ← proximity intel (if applicable)
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━  │
│  [ NEUTRALIZE ]        [ SKIP ]     │  ← action buttons
└─────────────────────────────────────┘
```

**THREAT CONFIDENCE** is always shown but is *not always reliable*. At low agent levels it drifts. At high levels it becomes a meaningful prior — but the player still needs the field data to update their estimate.

## THREAT CONFIDENCE — How It Works

The **Threat Confidence** bar is always visible and gives a starting prior. It is computed procedurally from the field draw — certain fields push it up or down, and the player can observe this over time.

Rules (player-learnable through play, never explained):

- Any **Category B** field with a suspicious value: +8–15% confidence
- **INTEGRATION DATE** before 1990: +20%
- **HUMAN FLUENCY SCORE** below 60%: +25%
- **COVER IDENTITY VERSION** above v5: +10%
- **SIGNAL BLEED = CRITICAL**: +30%
- **BODY TEMP** in normal range: −5% (mild reassurance)
- **SHADOW CONSISTENCY = VERIFIED**: −10%
- **DOES NOT DREAM**: +12%
- **CHILDHOOD (CONFIRMED?) = YES**: −8%
- Category C glitch fields: ±0% (pure noise — but players overthink them anyway)

The confidence bar is a *prior*, not a verdict. The player's job is to decide whether the fields justify tapping — or whether the confidence is itself being spoofed.

## CLUSTER SIGNAL — Proximity Layer

When two or more tagged contacts are within close physical proximity, the overlay shows:

```
CLUSTER SIGNAL:    ▲ +2 NEARBY
CLUSTER THREAT:    ██████░░  71%
```

This echoes Minesweeper's numbered cells. The cluster threat is an aggregate — it doesn't tell you *which* nearby contact is the alien. It tells you the density.

Design intent: the player has to decide whether to defuse the cluster (risky — touching one might detonate adjacents) or withdraw and scout from a different angle.

Cluster sizes:

| Size | Label | Effect |
|---|---|---|
| 2 | DYAD | Mild threat amplification |
| 3–4 | CELL | Moderate — at least 1 alien probable |
| 5+ | NEST | High — chain reaction possible |

A **chain reaction** occurs when neutralizing one alien in a NEST causes a brief cascade of AR debris from all adjacent aliens simultaneously — maximum chaos, maximum reward window, maximum scrambler risk.

## PLAYER-FACING FIELD LITERACY (Progression System)

Players do not receive a manual. They learn field meanings through play. The progression system surfaces this organically:

| Agent Level | Overlay Unlock |
|---|---|
| Recruit | 3–4 fields per scan, confidence bar only |
| Operative | Field history log (see past scans) |
| Field Agent | Highlight anomalous fields in amber |
| Analyst | Show which fields pushed confidence up/down |
| Senior Agent | Cross-session field frequency stats |
| Director | Custom field filters — suppress noise categories |

This means a **Recruit** reads an overlay like a wall of noise.
A **Director** reads it like a spreadsheet. Same data, different signal-to-noise ratio.

## TONE REFERENCE — UI COPY BIBLE

All system messages, briefing texts, and error states follow these rules:

- **Voice**: cold government bureaucracy. No exclamation marks. No warmth.
- **Tense**: present or past passive. "Contact has been flagged." Not "You flagged a contact."
- **Uncertainty is procedural, not apologetic.** "Confidence value unstable. Proceed at discretion." Not "Sorry, we're not sure."
- **Rare moments of almost-humanity**: a field that says "favorite color: beige" is funnier *because* nothing else in the UI has personality.

Sample system messages:

> *"Signal degraded. Field data may not reflect current operative status. This is noted in your record."*

> *"You have neutralized 2 contacts this session. 1 confirmed operative. 1 unresolved. The Resistance does not assign blame. It assigns probability."*

> *"DORMANT operatives do not count toward your quota. They count toward theirs."*

> *"Your error rate this week: 40%. Acceptable range: <30%. You have been reclassified as a liability. This is temporary."*

> *"Contact skipped. Contact has noted being skipped. This may not matter."*

## OPEN DESIGN QUESTIONS — To Resolve in Next Pass

1. **Field spoofing mechanic**: high-level aliens could have *deliberately falsified* overlays — all Category A data, confidence stuck at 15%. A countermeasure for expert players who have learned the field patterns. How do we surface this as a mechanic without making it just random again?

2. **The "perfect human" paradox**: an operative who scores 99.2% Human Fluency is more terrifying than one who scores 40%. Should extreme normalcy be a flag? How do we communicate this without a tutorial?

3. **Dynamic field frequency per session**: if the game shifts which fields appear based on difficulty tier, does this preserve the literacy arc or undermine it?

4. **Field localization/cultural variation**: some fields (library card, cheese preference) are inherently Western/quirky. Does the tone hold in other cultural contexts? Does it need to?

5. **The lie field**: one field per session is always false — generated by the system itself as a test of agent reliability. Does the player ever learn this? Do they ever learn *which* field?

6. **Multiplayer overlay divergence**: in a shared Field Run, two agents scanning the same contact could see *different overlays* — the Signal is personal, unreliable, subjective. Discuss.

## TECHNICAL CONSTRAINTS NOTE

The on-device model detects **person presence and rough posture/location** only. It does not identify, classify, or store any real-world individual data. All overlay fields are **procedurally generated at scan time** and seeded by:
- Session parameters (difficulty, alien count)
- Contact index within session
- Randomized field pool draw with category weights
- Player agent level (filters noise at high levels)

No real biometric or identity data is collected, stored, or inferred. The overlay is entirely fictional. This must be communicated clearly in onboarding.
